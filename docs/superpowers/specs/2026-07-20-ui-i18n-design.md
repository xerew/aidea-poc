# UI Internationalization (Phase 1) — Design

Approved decisions (user): 9 languages — English (en), Greek (el), French
(fr), Spanish (es), Italian (it), Finnish (fi), Swedish (sv), Norwegian
(no), German (de). Phase 1 = interface chrome only, static build-time
dictionaries. Phase 2 (course content, out of scope here) will use the
user's self-hosted Ollama over an autossh sidecar tunnel.

## Language field

`UserProfile.language` (CharField, 2-char, choices = the 9 codes, default
`en`). Exposed read-only on `UserSerializer.profile` (so `/auth/me/` and
login carry it) and writable via a new `ProfileLanguageView`
(`PATCH /api/profile/language/ {language}`) — any authenticated user.

**Default from country at registration:** `RegisterSerializer` derives
`language` from the submitted `country` via `COUNTRY_TO_LANGUAGE`
(backend `hub/i18n.py`), defaulting to `en` for unmapped countries. Always
overridable later in the profile.

`COUNTRY_TO_LANGUAGE` (primary mappings; everything else → en):
- el: GR, CY
- fr: FR, MC, LU
- es: ES, MX, AR, CO, CL, PE, VE, EC, GT, CU, BO, DO, HN, PY, SV, NI, CR, PA, UY
- it: IT, SM
- fi: FI
- sv: SE
- no: NO
- de: DE, AT, CH, LI

## Frontend (react-i18next)

- Deps: `i18next`, `react-i18next`, `i18next-browser-languagedetector`.
- `frontend/src/i18n.js`: initializes i18next with the 9 resources,
  `fallbackLng: 'en'`, `supportedLngs`, detection order
  `['localStorage', 'navigator']` (key `aidea_lang`). Provider wraps the
  app in `main.jsx`.
- One dictionary per language: `frontend/src/locales/<code>.json`, nested
  keys by area (`nav.*`, `auth.*`, `home.*`, `courses.*`, `lesson.*`,
  `authoring.*`, `reviews.*`, `profile.*`, `admin.*`, `common.*`).
  `en.json` is the source of truth; the other 8 are translated copies with
  identical key sets. Missing keys fall back to English automatically.
- Dynamic text uses i18next **interpolation** (`t('courses.count', {count})`)
  and plural keys where needed ("X of Y completed", "Question 1 of 3",
  "5 courses").
- **Language switcher**: a globe dropdown in the Header listing the 9
  languages (native names). On change: `i18n.changeLanguage(code)` +
  `localStorage['aidea_lang']` + (if logged in) `PATCH /profile/language/`.
- **Login/Register (anonymous)**: the same switcher writes only localStorage;
  the detector picks it up. On login, `AuthContext` calls
  `i18n.changeLanguage(user.profile.language)` so the account's stored
  language wins.

## Dictionary generation

`en.json` is extracted from the current hardcoded strings during
conversion. The 8 translated dictionaries are generated (Claude) from the
complete `en.json` in a dedicated task on a capable model, preserving
interpolation placeholders (`{{count}}`) verbatim. A key-parity test
asserts every language file has exactly the same key set as `en.json`.
Honest caveat: machine/AI translations — a native speaker should review
Greek/Finnish/Norwegian/etc. before production; acceptable for the POC.

## Out of scope (Phase 1)

Course content, quiz text, Django admin, transactional emails, RTL
languages, date/number locale formatting.
